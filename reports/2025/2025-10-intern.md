## [姚鑫鹏](../../Intern/intern_message.md#姚鑫鹏)

### 感悟思考

本月进行了一些 spec 文件的自动化思考，我现在提交 pr 的方法有点过于原始了，而且经常遇到自己不严谨的思考导致的一些细节上的问题，对此我做出了深刻的思考，我认为以我的脑袋思考来解决大部分的通用问题是很难提高工作效率的，因此我进行了 spec 文件的自动化编写的探索。正好接了一个调查一些发行版的特性研究任务，我认为其他发行版的自动化方案十分不错，或许我可以借鉴，比如在打包前阶段进行调研使用自动化工具查看构建依赖，和基础 spec 文件的信息，然后根据构建系统和构建参数快速生成 spec 文件，这样我们只需要关注这些软件的构建特性来补充里面的细节，这对于简单的软件打包来说可以极大程度的减少工作量，及其复杂的软件可能还是需要根据官方的构建文档和其他发行版的参考去补充。

目前在做的尝试

- 采用静态代码分析的方式来检查 spec 文件编写情况，比如文件编写的格式，我们可以自定义规范
- 在本地快速创建一个 spec 文件，但是需要根据需要构建的软件逆向推导完成，需要根据我们的文件标准进行设计，这样可以省去一部分工作
- 参考一些自动化工具来快速 git ，减少推送测试成本
