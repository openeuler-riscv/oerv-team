## 陈岳涛

### 感悟思考

这个月在前面调研的基础上，继续看 ROCm 在 RISC-V 上实际运行时会卡在哪些地方。因为我这边是远程参与，没有直接持有完整的 SG2044/ROCm 调试环境，所以主要是根据老师和同学同步的命令输出、日志、gdb 栈和 dmesg 信息来做研究性分析。实际感受是，上层算子和编译优化当然是更熟悉也更有意思的方向，但在硬件和 runtime 基线还没有稳定之前，直接讨论 kernel 调度或算子优化会缺少可靠参照；因此这个月更多是在把“算子生态能不能跑”往下拆到 Linux GPU runtime、驱动接口和运行时依赖这一层来理解。

### 外部可见交付

本月主要是把远程同步的复现记录按软件栈重新捋了一遍：设备能不能识别、OpenCL 平台能不能发现、HSA runtime 能不能支撑数据搬运和任务提交、内存一致性的假设是否成立。这样整理之后，后续讨论 RISC-V 上开源 GPU 计算生态支持度时，就不会只停留在某个命令能不能跑。

- 根据远程同步的 SG2044 等 RISC-V 环境运行结果，参与整理了几组典型现象：`rocminfo` 可以输出设备信息，但 `rocm-clinfo` 会在 `clGetPlatformIDs(-1001)` 处失败，`rocm-bandwidth-test` 则会触发 memory critical error 或后续的 SIGSEGV。这个结果至少说明 `rocminfo` 能输出设备信息并不等价于 OpenCL/HSA/实际拷贝链路已经完整跑通，因此可以把设备枚举、OpenCL 平台发现和 HSA copy/queue 执行视作不同层次的生态支持问题。
- 结合带宽测试、OpenCL 平台发现和 runtime 栈信息，不只看命令本身能不能跑，而是看 ROCm 在 RISC-V 上能不能稳定提供上层算子库和编译组件所需要的数据搬运、任务提交、事件同步和内存语义。
- 少量偏底层的线索没有继续摊开写，主要把它们放在 GPU runtime 对 Linux 内存映射、事件同步和一致性语义的依赖这个背景下理解。
- 根据远程环境里带调试信息的 ROCm runtime 和 bandwidth test 输出，保留了 HSA 异步拷贝、AQL queue 写入和相关内存属性这些关键词。它们对上层算子研究的意义在于：只有 runtime 的数据流转和任务提交路径足够清楚，后续讨论算子 kernel、图编译或编译优化才有比较可靠的基础。

### 正在进行的尝试/未来的计划

- 继续整理 ROCm 在 RISC-V 上的远程复现材料，把 `rocminfo`、`rocm-clinfo`、bandwidth test、gdb 栈和 dmesg 日志按技术层次归档，作为后续研究和讨论材料。
- 后续继续对比 CUDA 生态中的数据搬运、stream/event 和 ROCm/HSA 的 async copy、signal、AQL queue 机制，重点理解模型推理链路在不同 GPU runtime 设计下的依赖差异。
- 如果后续硬件和 runtime 基线更稳定，再围绕 DeepSeek 或其他大模型 demo 的最小推理样例继续往上层推进，把模型框架、算子库、图编译和编译优化重新接回到同一条生态链路里。
