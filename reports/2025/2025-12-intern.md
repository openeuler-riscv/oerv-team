## [明岢岢](../../Intern/intern_message.md#明岢岢)

### 感悟思考

本月主要是进行了一些探索吧.

<a id="target1"></a>

1. goblin: goblin 是一个 rust 编写的二进制文件解析软件, 当时在调研的时候看到了这个软件, 最初是因为作为依赖感觉引用量非常高(看了一下数据, github 显示已知的高达3w多个仓库), 于是我就关注了一下, 在查看仓库的时候发现 riscv 代码有一部分已经过时了, 于是我就修正了一下, 在跟作者的商讨下决定拆分成两个PR, 一个做增量性修改, 另一个做 breaking change 的修改(作者 approve 了, 他的描述是这些 breaking 的修改将在后期发版的时候再合并). 起初的修改是因为看到代码跟规范的文档有一小部分不对版, 就直接进行了修改, 没有想太多, 直到 maintainer 询问我为何进行一些 breaking change 的时候, 我仔细地翻阅了 riscv abi 修正过程中的一些 issue 和 PR, 认识到了一些规范为何要进行修改, 对 PR 进行了 defense, 此间学到了不少.

<a id="target2"></a>

2. angr: 正在跟进的部分有3个PR, 其中一个 PR 是在移植的时候发现他们 fuzzing 没覆盖的地方存在代码的错误, 于是就进行了修正. 另外2个部分是针对 riscv64 的移植部分(移植过程中发现了他们还不支持RVC, 于是我提了一个[RFC](https://github.com/angr/archinfo/issues/324), 貌似他们也意识到了这个问题, 最近进行了相关的修改, 昨日已经更新了), 还有1个PR虽然没有合并, 但是 maintainer 表示了 appreciate, 只是要我把杂项整理一下过 ci, 已经更改了但是还没看到 maintainer 的下一步动作. 余下1个PR已经合并, 该PR和我修改的上一个PR是强相关的. 该过程中的修正完全是围绕 riscv abi 展开的, 学到了很多东西.

<a id="target3"></a>

3. miasm: 该工具其实和 angr 功能有些相似, 于是我也在思考移植该工具的意义, 这是在初期进行移植的, 当时没有一个更多的思考, 但是已经支持了基本的指令集, riscv64 的框架也搭建好了, 余下的其实就是填充, PR 还没有合并, maintainer 表示在 15-20 天内帮忙 review, 到目前还没有动静, 但是该工具不会作为一个重要的移植点.


<a id="target4"></a>

4. gef: 这是一个 gdb 的插件, 跟很多的逆向软件都有拓展, 于是进行了一下尝试, 还挺好用的. 当然使用的过程中发现了一些问题, 于是也进行了修正(maintainer 审核过了, 已经根据建议进行修改, 然后就没看到下一步动作了).
