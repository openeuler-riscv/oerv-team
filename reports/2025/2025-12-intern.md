## [王尧勇](../../Intern/intern_message.md#王尧勇)

### 感悟思考

- 本月继续完成了 SP3 RVA23 的两个测试任务。
- 本月主要工作是尝试适配 Heolleo，遇到了 electron 相关的一系列问题。electron 官方并没有提供 riscv64 架构的支持包，因此需要自己制作好压缩包放在本地的下载缓存来绕过下载报错。一开始使用的是 openSUSE 的 electron 包，手动组织成 npm 包，发现缺少了太多依赖库；之后在 github 找到了打包好的二进制，可以正常使用并且没有依赖问题。
- 在构建 electron 应用时，发现 electron-builder 包也缺乏 riscv64 支持。首先是在 `arch.ts` 中缺少 riscv64 的定义和相关分支，手动添加即可解决。
- 另一个[问题](https://github.com/openeuler-riscv/oerv-team/issues/2101#issuecomment-3697358580)则更加棘手，发现在打包 asar 资源文件时，某些文件的 `metadata` 这个 map 对象的值被改变了类型，从表示文件路径的字符串变成了数字，而且所有出错的文件都变成了 63 这个数字。考虑到 electron-builder 是纯js脚本，那么最可能出错的点就是在读取文件时就已经出错了。然而经过检查，发现一开始读取的文件元信息都是正确的，是这个对象在函数之间传递时，值突然被改变了，并且每次出错的文件都不一样。

## [明岢岢](../../Intern/intern_message.md#明岢岢)

### 感悟思考

本月主要是进行了一些探索吧.

<a id="target1"></a>

1. goblin: goblin 是一个 rust 编写的二进制文件解析软件, 当时在调研的时候看到了这个软件, 最初是因为作为依赖感觉引用量非常高(看了一下数据, github 显示已知的高达3w多个仓库), 于是我就关注了一下, 在查看仓库的时候发现 riscv 代码有一部分已经过时了, 于是我就修正了一下, 在跟作者的商讨下决定拆分成两个PR, 一个做增量性修改, 另一个做 breaking change 的修改(作者 approve 了, 他的描述是这些 breaking 的修改将在后期发版的时候再合并). 起初的修改是因为看到代码跟规范的文档有一小部分不对版, 就直接进行了修改, 没有想太多, 直到 maintainer 询问我为何进行一些 breaking change 的时候, 我仔细地翻阅了 riscv abi 修正过程中的一些 issue 和 PR, 认识到了一些规范为何要进行修改, 对 PR 进行了 defense, 此间学到了不少.

<a id="target2"></a>

2. angr: 正在跟进的部分有3个PR, 其中一个 PR 是在移植的时候发现他们 fuzzing 没覆盖的地方存在代码的错误, 于是就进行了修正. 另外2个部分是针对 riscv64 的移植部分(移植过程中发现了他们还不支持RVC, 于是我提了一个[RFC](https://github.com/angr/archinfo/issues/324), 貌似他们也意识到了这个问题, 最近进行了相关的修改, 昨日已经更新了), 还有1个PR虽然没有合并, 但是 maintainer 表示了 appreciate, 只是要我把杂项整理一下过 ci, 已经更改了但是还没看到 maintainer 的下一步动作. 余下1个PR已经合并, 该PR和我修改的上一个PR是强相关的. 该过程中的修正完全是围绕 riscv abi 展开的, 学到了很多东西.

<a id="target3"></a>

3. miasm: 该工具其实和 angr 功能有些相似, 于是我也在思考移植该工具的意义, 这是在初期进行移植的, 当时没有一个更多的思考, 但是已经支持了基本的指令集, riscv64 的框架也搭建好了, 余下的其实就是填充, PR 还没有合并, maintainer 表示在 15-20 天内帮忙 review, 到目前还没有动静, 但是该工具不会作为一个重要的移植点.


<a id="target4"></a>

4. gef: 这是一个 gdb 的插件, 跟很多的逆向软件都有拓展, 于是进行了一下尝试, 还挺好用的. 当然使用的过程中发现了一些问题, 于是也进行了修正(maintainer 审核过了, 已经根据建议进行修改, 然后就没看到下一步动作了).

### 感悟思考

我总是想着能做点什么为社区，然后找一些自己可以做的工作，比如很简单的修包等操作，修修 bug，但是由于家里和学校里等各种原因，本来计划的工作时间被无限制的压缩，开源工作是一个非常需要坚持的事业。我看到群里的老师和同学的工作能力和效率令我遥不可及，我总是在想为什么我不能快速完成这么简单的任务，但是当你实际上要去做一些知识里知识这么‘简单’的任务时，你就知道，你还不够专业，你需要的是大量的积累，以及确定性的答案，而不是借助外力（比如问问 AI）。我总是在专业的知识中游荡，而没有真正的触及他们，把他们印在脑袋里，如果你每天从事一样的事情，再笨也可以上手。但是问题是我缺乏对于知识的敬畏和思考，导致我在追求完成工作中迷失了自我。

在这个大家庭中，老师们都很专业于自己的领域，每个人都能找得到自己要做的方向，我给自己的定位就是群里的混子，没事的时候看看群里的大佬谈论的问题我能否理解，尝试用自己的方式去考虑一下我能不能听懂，或者去看看这个方向自己有没有思路。

我目前做的最多的工作就是在 ruyi 的系统里大的包，但是我那时候考虑的不够多，很多的包都是照着 fedora 的版本比较然后一点点修改能成功我就会提交，但是各种细节上的问题并没有深入去思考。后来我由于个人原因将原先的包仔细检查之后提交剩余的包就没有再继续提交了，因为我认为如果我为了效率提交的包自己都不能维护或者理解，会对后续的维护者产生很大的误解。其他同学很快就能打完的包我可能需要一天两天，可能是我注意的方向不对，但是工作效率低下的本质就是对于这些问题的细节没有考虑清楚，如果你能完全确定这些问题，不用思考就可以完成一些操作，那么你的效率肯定能提高，但是我对于这些事总是有很多疑惑，比如如何快速定位 BuildRequires，直接借鉴其他系统的版本可能会导致包名不同，或者引入方式不同产生歧义，然后我们的编写方式也需要改变，可能这方面通过脚本快速提取会快很多（已经有同学做出来了，或许这就是高效率的原因）。

我知道我应该找到一条属于自己的维护方向，而不是继续在自己不擅长的领域里跟专业的同学相比。我总是想着把技术学会了再去努力工作，但是人们都是在工作中成长，我是在工作中抱怨自己学的还不够多。

最近两个月完成了家里安排的考试（虽然我感觉并没有什么用），但是如果我能摆脱这种管理可能我也不会到这个地步？我认为我是时候做出一些改变了，所以我在毕设的时候仔细思考了自己要做的方向，我打算从操作系统的学习开始，逐步的完成我对 Linux 方面的知识的总结，这样我在完成毕设的时候也能快速的让自己吸收这些专业知识，完成自己的需求，可能我会花大量的精力去完成这个任务，而且这些系统的设计也包含了我对 AI 智能对于未来学习方向的思考。如果我这个项目可以完成初步设计，我可以将我在 oerv 中学到的知识全都融合进去，也是可以让更多的人能够知道这些优秀的开源项目，让更多的人有能够提交贡献的能力。我感觉如果我可以为组织做一些贡献的方式也就在这了吧。
